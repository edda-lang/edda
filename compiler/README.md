# The Edda compiler

The native Edda compiler, written in Edda — a self-hosting ahead-of-time compiler that type-checks, discharges refinement proofs with an in-tree SMT solver, and emits native object code for x86-64, AArch64, and WebAssembly. ~178,000 lines of Edda across a 44-member workspace.

The compiler is bootstrapped: the binary that builds it is the [Rust bootstrap compiler](https://github.com/edda-lang/edda-bootstrap) (the reference implementation); the native compiler compiles the same `.ea` source through its own backend, with full self-compilation gated today by the compiler's performance — chiefly the memory it needs to build its heaviest members. The language reference lives in [`../codex/language/`](../codex/language/).

## Pipeline

Source flows through the workspace members in this order:

```
parse → resolve → lower(HIR) → cteval → typecheck → specs(monomorphise)
      → refine(SMT discharge) → lower(MIR) → optimize → emit → link
```

- **Frontend** — `syntax` (lexer/parser), `resolve` (name + capability resolution), `hir` + `lower` (desugaring), `types` (inference, effect-row checking), `cteval` (comptime evaluation).
- **Generics** — `specs`: Edda has no `<T>` generics or traits; reuse is `spec` templates, content-addressed and monomorphised per instantiation.
- **Verification** — `refine` lifts `requires`/`ensures`/`decreases` and built-in obligations (overflow, bounds, division) to SMT and discharges them with its own in-tree solver: a CDCL SAT core (`solver/cdcl`) with Tseitin encoding, driving a DPLL(T) combination of EUF (congruence closure) and linear integer arithmetic theories. `verify` checks MIR boundary invariants.
- **Backend** — `mir` (mid-level IR), `optimize` (e-graph rewriting via `egraph`, refinement-driven trap elimination), `regalloc` (linear-scan, x86-64 + AAPCS64), the `x64` / `arm64` / `wasm` code generators, `object` (COFF / ELF / Mach-O emission), and `link` (drives `lld-link`, `ld`, `ld64.lld`, `wasm-ld`).

Supporting members: `target` (triple / arch / OS / ABI / capability tables), `diagnostics`, `source`, `span`, `intern`, `query` (incremental query graph), `cache` (content-addressed artifacts), `manifest` (`package.toml`), `structmap` (emits the `index.toon` structure maps), `debug` (line tables / locals), `inspect` (per-pass IR dumps), the `driver` / `edda` CLI members, and the `*_verify` / `*_goldens` differential-verification harnesses.

## Targets

| Arch | OS | ABI |
|---|---|---|
| `x86_64`, `aarch64`, `riscv64`, `wasm32`, `wasm64` | `linux`, `windows`, `macos`, `freebsd`, `bare_metal`, `wasi`, `browser` | `gnu`, `musl`, `msvc`, `darwin`, `wasi_preview1`, `none` |

Default target: `x86-64-windows-msvc`. Code generators exist for x86-64, AArch64, and WebAssembly; per-target capability availability (e.g. `Subprocess` and `Debugger` only on hosted OSes) is enforced by the `target` member.

## Building

The compiler is built by the bootstrap toolchain, not `cargo`. Build the bootstrap once (its README carries the steps), then run its `edda` binary against this workspace:

```bash
edda build                                  # type-check + refinement discharge + codegen
edda check                                  # type-check only
```

`edda build` and `edda check` operate on the workspace; `edda version [--json]` reports the compiler identity. The native driver implements `build` / `check` / `run` / `test` / `version`; other subcommands (`bench`, `fmt`, `lint`, `gc`, `daemon`, `structmap`, …) are parsed by the CLI but carried by the bootstrap for now.

## Layout

```
lib/<member>/src/*.ea   the 44 workspace members
../std/                 the std.* standard library
../runes/               ecosystem packages
../web/                 first-party Edda applications
../codex/               the language specification
```

`index.toon` files are compiler-emitted structure maps (signatures, effect rows, refinements, call graph), regenerated by every `edda build`. Edda source admits no comments; structure is derived, not authored.
